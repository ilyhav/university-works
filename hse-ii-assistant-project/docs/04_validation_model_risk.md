# 04. Валидация и модельный риск

Документ относится к **критерию 4** проекта «ВкладНавигатор» (модуль *подбор вклада*).
Архитектура гибридная: детерминированное **ядро** — единственный источник истины по всем
цифрам, а **LLM-слой** (GigaChat-2-Max) только объясняет решение и не вводит новых чисел
(офлайн-режим без ключа). Здесь описано, как мы *проверяем* качество (часть А) и как
*управляем* модельным риском (часть Б). Смежные документы: архитектура — `docs/00_overview.md`,
метрики и мониторинг — `docs/03_monitoring.md`, регламент вывода модели — `docs/06_decommission.md`.

## А. Валидация на golden-наборе

### Golden-набор и прогон

Эталонный набор `data/golden_set.json` содержит **17 кейсов**: **9** с ожидаемой выдачей
(`ok`) и **8** с ожидаемым отказом (`refused`). Прогон выполняет
`src/validation/golden.py::evaluate_golden`: для каждого кейса вызывается
`recommend(...)`, а функция `_case_passed` сверяет статус, код отказа, идентификатор top-1,
жёсткие ограничения (`expected_constraints`) и наличие нужного риск-флага. Запуск
воспроизводим командой `make eval` (`scripts/run_eval.py`).

### Метрики качества

Метрики считаются в `src/validation/golden.py::evaluate_golden`, опираясь на чистые
функции из `src/validation/metrics.py` (`Confusion`, `precision_recall_f1`,
`reciprocal_rank`, `ratio`):

| Группа | Что в отчёте | Как считается |
|---|---|---|
| Ранжирование | `top1_accuracy`, `mrr` | `ratio()` и `reciprocal_rank()` в `golden.py` (эталон — доходный оптимум) |
| Корректность отказов | `precision`, `recall`, `f1` | `precision_recall_f1(Confusion)` в `metrics.py` |
| Точность кода отказа | `code_accuracy` | `ratio()` в `golden.py`: совпадение кода (из 8) с эталоном |
| Ограничения | `constraints.rate` | `ratio()` в `golden.py`: жёсткие условия клиента в top-1 |
| Groundedness LLM | `check_groundedness` | `src/llm/groundedness.py`: доля чисел в тексте, обоснованных ядром |

Класс `Confusion` трактует отказ как «положительный» класс: `tp` — ждали отказ и отказали,
`fp` — отказ при ожидаемой выдаче, `fn` — выдача при ожидаемом отказе, `tn` — корректная
выдача. **Groundedness** собирает закрытый список разрешённых чисел (`allowed_numbers`:
ставки, суммы, лимит АСВ, свежесть и т. п.), извлекает все числа из текста модели и помечает
любое «новое» число как `ungrounded`; `score == 1.0` означает, что модель не придумала ни
одной цифры. В `explainer.py` порог `ACCEPT_THRESHOLD=1.0`, поэтому при необоснованном
числе ответ откатывается к детерминированному шаблону.

### Фактические результаты (`make eval`, воспроизводимо)

- Пройдено **17/17** кейсов; `pass_rate = 1.0`.
- Ранжирование: **top-1 accuracy = 1.0**, **MRR = 1.0**.
- Отказы: **precision = recall = F1 = 1.0**; матрица ошибок `tp=8, fp=0, fn=0, tn=9`.
- Точность кода отказа = **1.0** (все 8 кодов попали в нужный класс).
- Проверка ограничений: top-1 удовлетворяет жёстким условиям во всех проверяемых кейсах.
- Задержка ядра `p95 ≈ 0,1 мс`. **Вердикт эксплуатации: OK.**

### Почему эталон top-1 не тавтологичен

Ключевой момент: **эталонный лучший вклад вычисляется независимо от скоринга**. Для цели
«максимум дохода» аналитический максимум определяется прямым перебором отфильтрованных
вкладов по эффективной доходности (APY с капитализацией, `src/core/effective_rate.py`) —
это закрытая формула, а не результат ранжирующей модели `ranking.py`
(`RANKING_VERSION="ranking-1.0.0"`, `GOAL_WEIGHTS`). Поэтому сравнение «top-1 == доходный
оптимум» проверяет *внешнюю* правильность, а не сходимость модели с самой собой. Совпадение
`top-1 = 1.0` означает, что политика скоринга согласована с экономически верным ответом, а
не подогнана под него.

В терминах курса это **ранжирование по детерминированным формулам, а не причинный вывод**:
причинность (Athey & Imbens, 2019) уместна при оценке *бизнес-эффекта* (A/B-тест влияния на
конверсию), а не при выборе вклада. Ядро ближе к прозрачной модели правил (объяснимость как
требование регулятора), LLM — генеративный слой UX вне критического пути решения (подход
*ML-solution-driven*, а не *model-driven*).

## Б. Модельный риск и ML-специфичная хрупкость (MRM)

Ключевая особенность ML-систем — они ломаются **не крашами, а тихо**: модель уверенно выдаёт
правдоподобно-неверное (галлюцинация), качество деградирует под дрейфом данных, ответы
портятся на входе вне обучающего распределения. Обычное тестирование такие режимы не ловит.
Главная мера митигирования у нас **архитектурная**: решение принимает детерминированное ядро,
а ML (LLM) вынесен из критического пути — большая часть ML-риска убрана из контура решения.
Остаточная ML-хрупкость живёт в LLM-слое и в дрейфе данных, и под каждый её вид есть
наблюдаемая метрика (детект) и конкретное действие (митигирование) — таблица ниже.

Управление модельным риском опирается на ЦБ РФ «Обзор практик применения ИИ в финансовом
секторе» (2025), а также SR 11-7 и практики KPMG/Deloitte, где источники риска делятся на
**данные / модель / использование**. Соответствие источников риска и мер контроля:

| Источник риска (категория MRM) | Как проявляется | Мера контроля |
|---|---|---|
| **Устаревшие данные** (данные) | каталог `data/deposits.csv` отстаёт от рынка, выдача неактуальна | мониторинг свежести (`drift.py`), `freshness_warning=14`, `critical=28`; вердикт `REFRESH` при превышении (см. `docs/06_decommission.md`) |
| **Ошибка формулы APY** (модель) | неверная эффективная ставка → неверный доход и ранжирование | золотые тесты на `effective_rate.py`, эталонный пример: номинал 17,9% + ежемесячная капитализация → **APY 19,44%**; независимый доходный эталон в `make eval` |
| **Галлюцинация LLM** (модель/использование) | модель называет несуществующее число | `check_groundedness` + `ACCEPT_THRESHOLD=1.0`; откат к шаблону; при срыве доли — действие **LLM_OFF** (ядро остаётся), а не полный вывод |
| **Prompt injection** (использование) | попытка переопределить инструкции через свободный текст | `guards.check_scope` → код `PROMPT_INJECTION`; LLM вне критического пути |
| **Мисселинг** (использование) | подбор под завышенное ожидание доходности | `guards.check_expectation`: при `expected_rate > market_max + 3.0` → код `UNREALISTIC_EXPECTATION` |
| **Сумма вне АСВ** (использование) | клиент превышает лимит страхования 1 400 000 ₽ | расчёт `insured/uninsured_amount`, риск-флаг АСВ; лимит в `config.ASV_INSURANCE_LIMIT` |
| **Смещение в данных** (данные) | перекос архетипов/банков искажает ранжирование | контроль распределений по PSI (`drift.py`, порог 0,25); каталог из вымышленных банков, надбавки прозрачны |

Восемь кодов отказа (`INVALID_INPUT`, `OUT_OF_SCOPE`, `UNREALISTIC_EXPECTATION`,
`AMOUNT_OUT_OF_RANGE`, `TERM_UNAVAILABLE`, `CURRENCY_UNAVAILABLE`, `CONSTRAINTS_UNAVAILABLE`, `PROMPT_INJECTION`)
делают «корректный отказ» отдельным измеримым классом ответа. Так как для решений есть
безопасный детерминированный fallback, материализация LLM-рисков ведёт к **LLM_OFF**, а
не к выводу модуля; полный `DECOMMISSION` — крайняя мера при деградации самого ядра.
Регламент действий (`NONE → MONITOR → REFRESH → LLM_OFF → DECOMMISSION`) и пороги
`THRESHOLDS` описаны в `docs/06_decommission.md`, а сценарий дрейфа рынка (ключевая ставка
16,0 → 11,0, PSI ≫ 0,25, свежесть 107 дней → `REFRESH` → после обновления top-1 = 1.0) —
в `docs/03_monitoring.md`.

**Итог.** Валидация подтверждает корректность ядра (17/17, top-1/MRR = 1.0, P/R/F1 = 1.0
с независимым доходным эталоном), а матрица модельного риска показывает, что каждый
источник риска данных/модели/использования закрыт детерминированной мерой контроля и
наблюдаемой метрикой.
