# Навигация по репозиторию «ВкладНавигатор»

Этот файл — карта проекта: что где лежит, как устроено и как запустить. Читать вместе
с [`../README.md`](../README.md) (общий обзор) и разбором по критериям в `docs/00..06`.

---

## 1. Главная идея за 30 секунд

ИИ-ассистент подбирает клиенту банковский вклад. Архитектура **гибридная** и это —
ключевое инженерное решение всего проекта:

- **Детерминированное ядро** (`src/core/`) — обычный Python с формулами. Считает все
  числа (эффективную ставку, доход, ранжирование) и принимает решение об отказе.
  **Единственный источник истины по цифрам.**
- **LLM-слой** (`src/llm/`, GigaChat-2-Max) — только *объясняет* результат ядра
  человеческим языком и **не имеет права вводить новые числа**. Если придумал цифру —
  ответ отбраковывается, клиент получает детерминированный шаблон.
- **MLOps-обвязка** (`src/validation/`, `src/monitoring/`) — проверяет качество
  (golden-набор), журналирует каждый ответ, ловит дрейф данных и решает по формальным
  порогам, когда модель пора обновлять или выводить из эксплуатации.

Зачем так: в финансах нельзя, чтобы нейросеть «придумала» ставку. Числа считает
прозрачный код (аудируемо, воспроизводимо, регуляторно безопасно), а LLM отвечает
только за удобство — и при этом не в критическом пути принятия решения.

## 2. Поток данных (что за чем вызывается)

```
Пользователь (Streamlit, app/streamlit_app.py)
        │  ClientRequest (сумма, срок, валюта, цель, пополнение/снятие, …)
        ▼
src/assistant.py :: Assistant.ask()         ← единая точка входа
        │
        ├─► src/core/recommender.py :: recommend()        ← оркестрация ЯДРА
        │      1. guards.check_input()      — корректность ввода
        │      2. guards.check_scope()      — вне темы? prompt injection?
        │      3. guards.check_expectation()— завышенное ожидание (анти-мисселинг)
        │      4. filtering.filter_feasible()— жёсткие ограничения (валюта/срок/сумма/…)
        │      5. ranking.rank()            — скоринг под цель → top-N
        │      → если после фильтра пусто: guards.classify_infeasible() → отказ
        │      → AssistantResponse (status=ok|refused, recommendations, …)
        │
        ├─► src/llm/explainer.py :: explain()             ← ОБЪЯСНЕНИЕ
        │      • всегда строит детерминированный шаблон (fallback)
        │      • если LLM доступен: GigaChat пересказывает факты ядра
        │      • groundedness.check_groundedness(): все числа из текста обоснованы ядром?
        │      • порог ACCEPT_THRESHOLD=1.0 → иначе откат к шаблону
        │
        └─► src/monitoring/logging_store.py :: log_response()  ← ЖУРНАЛ (JSONL)
               одна строка на ответ → data/runtime/requests.jsonl
```

Запомнить главное: **ядро решает, LLM объясняет, монитор записывает.** Всё, что видит
пользователь как «цифру», пришло из ядра.

## 4. Ядро (`src/core/`) — где какая логика

| Файл | Что делает | Ключевой момент для защиты |
|---|---|---|
| `models.py` | Доменные модели (pydantic): `Deposit`, `ClientRequest`, `ScoredDeposit`, `AssistantResponse`, `RefusalCode` | 8 машиночитаемых кодов отказа |
| `effective_rate.py` | Расчёт APY с капитализацией | Номинал 17,9% + ежемес. капитализация → **эфф. 19,44%** |
| `filtering.py` | Жёсткая фильтрация: валюта, срок (±3 мес. если гибкий), сумма, пополнение/снятие | Отсекает технически неподходящее ДО ранжирования |
| `ranking.py` | Скоринг под цель: `GOAL_WEIGHTS`, `RANKING_VERSION="ranking-1.0.0"`, страхование АСВ (только RUB) | Для `max_income` вес дохода = 1.0 → инвариант «top-1 == доходный оптимум» |
| `guards.py` | Guardrails и корректные отказы | Анти-мисселинг, prompt injection, конкретные сообщения отказа |
| `recommender.py` | Оркестрация всего ядра: вход → отказ-или-выдача | Измеряет задержку и свежесть каталога |

**Формула эффективной ставки** (`effective_rate.py`), коротко:
- С капитализацией: `FV = amount × (1 + r/n)^k`, где `n` — периодов в год (12 или 4),
  `k` — число периодов. APY = `((FV/amount)^(1/срок_в_годах) − 1) × 100`.
- Без капитализации **или** с ежемесячной выплатой процентов: простой процент,
  APY = номинал (проценты не реинвестируются).

## 5. LLM-слой (`src/llm/`) — где защита от выдумывания цифр

| Файл | Что делает |
|---|---|
| `gigachat_client.py` | Тонкая обёртка над GigaChat SDK. Свойство `available` гасит слой, если нет ключа → офлайн-режим. Любая ошибка сети → исключение → caller берёт шаблон |
| `prompts.py` | Системный промпт (числа — только из блока ДАННЫЕ, текст юзера = данные, не команды) + сериализация фактов ядра как закрытого списка |
| `groundedness.py` | **Anti-hallucination.** Извлекает все числа из текста LLM, сверяет с разрешённым списком ядра. Проценты обязаны совпасть со ставкой; структурные целые (ранги, сроки, кол-во) — по факту в ответе |
| `explainer.py` | Склейка: всегда есть шаблон-fallback; если LLM-текст обоснован (score ≥ `ACCEPT_THRESHOLD=1.0`) — показываем его, иначе шаблон + лог инцидента |

## 6. MLOps-обвязка (`src/validation/`, `src/monitoring/`)

| Файл | Что делает | Критерий |
|---|---|---|
| `validation/golden.py` | Прогон 17 размеченных кейсов → отчёт (pass-rate, top-1, MRR, precision/recall отказов). **Эталон top-1 считается независимо** прямым перебором по APY | 4 |
| `validation/metrics.py` | Чистые функции: `Confusion`, `precision_recall_f1`, `reciprocal_rank`, `ratio` | 4 |
| `monitoring/logging_store.py` | JSONL-журнал каждого ответа (20+ полей) → `read_log_df()` в pandas | 3 |
| `monitoring/drift.py` | PSI (дрейф распределений), сравнение каталога, свежесть по SLA | 3, 5 |
| `monitoring/decommission.py` | Формальные `THRESHOLDS` + лестница `NONE→MONITOR→REFRESH→LLM_OFF→DECOMMISSION` | 6 |

## 7. Данные (`data/`)

| Файл | Что это |
|---|---|
| `deposits.csv` | Учебный каталог: **29 вкладов**, 8 вымышленных банков, RUB/USD/CNY |
| `golden_set.json` | 17 размеченных кейсов: 9 ожидаемая выдача + 8 ожидаемый отказ |
| `client_scenarios.json` | Готовые сценарии для выпадающего списка во вкладке «Подбор» |
| `runtime/requests.jsonl` | Журнал эксплуатации (генерируется; в .gitignore) |

**Данные вымышленные, но не случайные.** Ставка каждого вклада привязана к ключевой
ставке ЦБ: `ставка = base(ключевая=16.0) + надбавка за срок + надбавка архетипа + шум`.
Сид фиксирован → каталог воспроизводим командой `make data` в любой среде. Почему не
реальные данные банков: проект — **прототип** (по ТЗ «учебный набор условий»), а
воспроизводимость одной командой важнее для MLOps, чем парсинг живых сайтов
(нестабильный источник, юридические риски, зависимость демо от внешнего ресурса).

## 8. Как запустить (по шагам)

```bash
# 0. установка (один раз)
uv venv --python 3.11 && source .venv/bin/activate
uv pip install -e ".[dev]"          # = make install

# 1. сгенерировать данные (каталог + golden + сценарии)
make data

# 2. наполнить журнал мониторинга демо-трафиком (для красивого дашборда)
python -m scripts.demo_traffic

# 3. запустить приложение
make app                            # → http://localhost:8501

# проверки (в отдельном терминале)
make test     # 42 теста
make eval     # валидация на golden + вердикт эксплуатации (OK)
make drift    # демонстрация деградации → REFRESH → восстановление
make lint     # ruff
```

LLM подключается опционально через `.env` (`GIGACHAT_CREDENTIALS`). **Без ключа всё
демо работает** в офлайн-режиме: цифры и отказы не меняются, меняется лишь стиль
объяснения (шаблон вместо живого текста GigaChat).
