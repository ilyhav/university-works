# 06. Критерии вывода модели из эксплуатации (decommission)

Документ описывает **критерий 6** проекта «ВкладНавигатор» (модуль *Подбор вклада*):
формальные условия, при которых модель частично или полностью выводится из
промышленной среды. Решение принимается по порогам, а не «на глаз», что соответствует
требованиям управления модельным риском (MRM: SR 11-7, KPMG, Deloitte; ЦБ РФ «Обзор
практик применения ИИ в финансовом секторе», 2025). Реализация — `src/monitoring/decommission.py`,
функция `evaluate(snapshot)`; входной снапшот собирается из `src/monitoring/logging_store.py`
(JSONL) и `src/monitoring/drift.py` (PSI, свежесть). Смежные документы: `docs/04_validation_model_risk.md`
(метрики), `docs/05_degradation.md` (дрейф), `docs/00_overview.md` (ядро vs LLM-слой).

## Архитектурный контекст: два масштаба вывода

Решение принимает **детерминированное ядро** (`src/core/recommender.py`) — оно источник
истины по всем цифрам и единственный элемент критического пути. **LLM-слой**
(GigaChat-2-Max, `src/llm/explainer.py`) лишь объясняет результат и не вводит новых чисел;
при отсутствии ключа он деградирует в шаблонный офлайн-режим. Это решение
*ML-solution-driven*: критический риск сосредоточен в прозрачном ядре, а генеративный слой
выведен из контура принятия решения. Поэтому отказ LLM по качеству приводит лишь к
**частичному выводу** (`LLM_OFF`) с безопасным fallback, а **полный вывод**
(`DECOMMISSION`) наступает только при отказе самого ядра.

## Пороги (`THRESHOLDS`)

Пороги версионируются вместе с моделью. Значения — из кода (`decommission.THRESHOLDS`):

| Метрика | Норма (`min`/`max`) | Критический порог | Источник |
|---|---|---|---|
| `groundedness_share` (доля обоснованных ответов LLM) | ≥ 0.98 | < 0.90 | `src/llm/groundedness.py` |
| `refusal_precision` (точность отказов) | ≥ 0.90 | < 0.70 | `src/validation/metrics.py` |
| `refusal_recall` (полнота отказов) | ≥ 0.85 | — | `src/validation/metrics.py` |
| `ranking_top1` (точность top-1) | ≥ 0.90 | < 0.75 | `src/core/ranking.py` |
| `latency_p95_ms` (задержка p95) | ≤ 1500 мс | — | `src/monitoring/logging_store.py` |
| `freshness_days` (возраст каталога) | ≤ 14 дн. (warning) | > 28 дн. | `src/monitoring/drift.py` |

Текущие значения валидации (`make eval`): groundedness обоснован, отказы
precision/recall/F1 = 1.0, top-1 = 1.0, p95 ядра ≈ 0,1 мс — итоговый вердикт **OK**.

## Лестница действий

Каждый критерий возвращает действие; итоговый вердикт — **самое серьёзное** из них
(`_ACTION_RANK`, `max`):

| Действие | Масштаб | Когда |
|---|---|---|
| **NONE** | Штатная работа | Все метрики в норме |
| **MONITOR** | Усиленное наблюдение | Рост доли необоснованных ответов; p95 > 1500 мс |
| **REFRESH** | Обновление данных/конфига | Каталог устарел (> 14/28 дн.); просадка top-1; качество отказов ниже нормы |
| **LLM_OFF** | Частичный вывод LLM-слоя | `groundedness_share` < 0.90: галлюцинации LLM; ядро остаётся, fallback включён |
| **DECOMMISSION** | Полный вывод модуля | `refusal_precision` < 0.70 **или** `ranking_top1` < 0.75: отказ ядра по качеству |

Ключевое различие: `LLM_OFF` снимает только генеративный слой (объяснения переключаются
на детерминированные шаблоны, цифры не страдают), тогда как `DECOMMISSION` останавливает
весь модуль, потому что подорвано само ядро принятия решения.

## Runbook

**Что триггерит.** Снапшот мониторинга агрегируется автоматически (логи + дрейф);
`evaluate()` сравнивает его с порогами и формирует `action` и `verdict`. Сигнал
отображается на вкладке *Мониторинг* в `app/streamlit_app.py` и воспроизводится скриптами
`scripts/simulate_drift.py` / `scripts/run_eval.py`.

**Кто принимает решение.** Триггер технический, но санкция на `LLM_OFF` и `DECOMMISSION` —
человеческая: **владелец продукта** (бизнес-ответственность) совместно с **валидатором
модельного риска** (независимая проверка по MRM). `MONITOR`/`REFRESH` выполняются в рабочем
порядке владельцем продукта; `DECOMMISSION` фиксируется протоколом с обоснованием.

**Порядок отката и ручной режим.**
1. `LLM_OFF` — выключить генеративный слой (убрать ключ / флаг); ядро продолжает выдавать
   подбор и объяснения по шаблонам. Деградация без простоя.
2. `DECOMMISSION` — отключить модуль *Подбор вклада*; в UI показать заглушку и направить
   пользователя в ручной режим (консультант банка / официальный калькулятор АСВ-лимита
   1 400 000 ₽). Зафиксировать причину и снапшот метрик.
3. Откат версии — вернуться к предыдущей `RANKING_VERSION` или предыдущему каталогу
   `data/deposits.csv` (см. `docs/05_degradation.md`: после `REFRESH` top-1 восстановился до 1.0).

**Условия повторного ввода в эксплуатацию.** Возврат допускается, когда: причина устранена;
прогон `make eval` снова даёт 17/17, top-1 ≥ 0.90, отказы precision ≥ 0.90 / recall ≥ 0.85,
groundedness ≥ 0.98 (для возврата LLM), свежесть ≤ 14 дн.; и валидатор модельного риска
подтверждает отчёт. Только тогда вердикт `evaluate()` возвращается к **OK**, и владелец
продукта повторно вводит модуль в промышленную среду.
