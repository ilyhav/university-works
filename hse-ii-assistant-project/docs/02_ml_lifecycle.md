# 02. Жизненный цикл ML-модели (ЖЦМ) как MVP

Документ описывает реализацию полного жизненного цикла ML-решения для модуля
**ПОДБОР ВКЛАДА** проекта «ВкладНавигатор» — ИИ-ассистента потребителя финансовых
услуг. Архитектура гибридная: детерминированное **ядро** (`src/core/`) — единственный
источник истины по всем числам, а **LLM-слой** (`src/llm/`, GigaChat-2-Max) только
объясняет результат и не вводит новых цифр. Смежные материалы: бизнес-ценность —
`docs/01_business_value.md`, мониторинг — `docs/03_monitoring.md`, валидация и метрики —
`docs/04_validation_model_risk.md`, вывод из эксплуатации — `docs/06_decommission.md`.

## Стадии жизненного цикла

**[1] Данные.** `scripts/catalog_builder.py` детерминированно строит каталог из
**29 вкладов**: `rate = base(key_rate) + надбавка за срок + надбавка архетипа`, где
базовая ключевая ставка ЦБ — `16.0`, дата каталога = дата сборки (`make data`).
`scripts/generate_dataset.py`
выгружает результат в `data/deposits.csv` и фиксирует эталон `data/golden_set.json`
(**17 кейсов: 9 ok, 8 refused**). Сид фиксирован → каталог воспроизводим. Банки
вымышленные. Пример скрытого условия: номинал 17,9% + ежемесячная капитализация →
эффективная APY 19,44% (`src/core/effective_rate.py`).

**[2] «Обучение».** Здесь нет градиентного спуска: «обучение» — это настройка и
фиксация **конфига скоринга**. Веса композитного ранжирования `GOAL_WEIGHTS` (по целям
`max_income`, `flexible`, `short_term`, `capital_protection`) и версия
`RANKING_VERSION = "ranking-1.0.0"` в `src/core/ranking.py` образуют версионируемую
«модель». Ранжирование смешивает доходность, гибкость, защищённость и ликвидность с
весами, зависящими от цели клиента.

**[3] Валидация.** `src/validation/golden.py` и `src/validation/metrics.py` прогоняют
конфиг на golden-наборе (`make eval`, воспроизводимо). Результат: пройдено **17/17**;
ранжирование top-1 = **1.0**; MRR = **1.0**; по отказам precision/recall/F1 = **1.0**;
точность кода отказа = **1.0**; confusion **tp8/fp0/fn0/tn9**; задержка p95 ядра
≈ **0,1 мс**. Вердикт эксплуатации — **OK**.

**[4] Развёртывание.** Интерфейс — Streamlit (`app/streamlit_app.py`, вкладки
Подбор / Качество / Мониторинг / О модели). `Dockerfile` собирает образ на
`python:3.11-slim`, ставит зависимости из `pyproject.toml`, копирует `src/`, `app/`,
`scripts/`, `data/`, открывает порт **8501** и добавляет HEALTHCHECK на `/healthz`.
Локальный запуск — `make app`.

**[5] Эксплуатация и журналирование.** Каждый ответ ядра пишется в JSONL через
`src/monitoring/logging_store.py`, включая `RANKING_VERSION` (управление моделью).
LLM-слой деградирует в офлайн-режим без ключа; `src/llm/groundedness.py` и
`explainer.py` (порог `ACCEPT_THRESHOLD=1.0`) откатывают к шаблону при необоснованном
числе. Защита включает 8 кодов отказа: `INVALID_INPUT`, `OUT_OF_SCOPE`,
`UNREALISTIC_EXPECTATION`, `AMOUNT_OUT_OF_RANGE`, `TERM_UNAVAILABLE`,
`CURRENCY_UNAVAILABLE`,
`CONSTRAINTS_UNAVAILABLE`, `PROMPT_INJECTION`.

**[6] Мониторинг.** `src/monitoring/drift.py` считает PSI по ставкам и свежесть
каталога (`make drift`, `scripts/simulate_drift.py`). Сценарий дрейфа: ключевая ставка
падает до **11.0** (−5 п.п.), промо-надбавка −3, max +0.5. PSI ≫ **0,25** (значимый),
MAE по ставке ≈ **5,4 п.п.**, точность top-1 относительно актуального рынка падает
**1.0 → 0.8**, свежесть **107 дней (critical)**.

**[7] Переобучение / обновление и вывод из эксплуатации.** Вердикт мониторинга —
**REFRESH**: после пересборки каталога (стадия [1]) top-1 снова **1.0**, вердикт **OK** —
цикл замыкается. Пороги вывода в `src/monitoring/decommission.py` (`THRESHOLDS`):
`groundedness_share_min=0.98 / critical=0.90`; `refusal_precision_min=0.90 /
critical=0.70`; `refusal_recall_min=0.85`; `ranking_top1_min=0.90 / critical=0.75`;
`latency_p95_ms_max=1500`; `freshness_warning=14 / critical=28`. Эскалация по
нарастанию: **NONE → MONITOR → REFRESH → LLM_OFF → DECOMMISSION**. Благодаря безопасному
детерминированному fallback галлюцинации LLM приводят к **LLM_OFF** (отключить LLM,
оставить ядро), а не к полному выводу модуля.

## Почему «модель» — это версионируемый конфиг правил

В критическом пути решения стоит не нейросеть, а **детерминированное ядро правил**.
«Обучаемая» часть — конфиг `GOAL_WEIGHTS` с явной семантикой и меткой
`RANKING_VERSION`. Это осознанный выбор в пользу **ML-solution-driven** подхода
(решение принимает прозрачное ядро, LLM не в критическом пути → управление сложностью),
а не **ML-model-driven**. По характеру задача — это ранжирование по детерминированным
формулам, а не причинный вывод; причинность (A/B, эффект на конверсию) уместна лишь при
оценке бизнес-эффекта (Athey & Imbens, 2019). Ядро ближе к прозрачной модели правил
(объяснимость как требование регулятора), тогда как LLM — генеративный слой UX.

Такой выбор снижает **модельный риск** (ЦБ РФ «Обзор практик применения ИИ в
фин. секторе», 2025; MRM-подходы KPMG, Deloitte, SR 11-7) по трём источникам:
**данные** — каталог воспроизводим и версионирован, дрейф детектируется по PSI и
свежести; **модель** — веса интерпретируемы, версионируемы и проверяемы на golden-наборе
(17/17), нет непрозрачных параметров; **использование** — каждый числовой вывод
обоснован ядром, LLM не вводит цифр, а при необоснованном числе срабатывает откат к
шаблону или `LLM_OFF`. Итог: предсказуемость, аудируемость и наличие безопасного
fallback — ключевые свойства управляемого модельного риска в финансовом домене.
