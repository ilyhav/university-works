# 00. Обзор проекта и его место в теории курса

Документ-навигатор для проекта **«ВкладНавигатор»** — ИИ-ассистента потребителя
финансовых услуг, модуль **подбор вклада**. Здесь зафиксированы задача, архитектурное
решение и теоретический контекст курса ВШЭ «Прикладное применение методов ИИ».
Разбор по критериям оценки 1–6 вынесен в соседние документы (карта — в разделе 6).
Все цифры — из кода и `make eval`/`make drift`; см. также корневой [`README.md`](../README.md).

## 1. Задача и образ результата

Клиент задаёт параметры (сумма, срок, валюта, цель, пополнение/снятие, ожидаемая ставка,
свободный вопрос), а ассистент помогает **принять информированное решение о вкладе**:
строит сравнительную таблицу top-N с расчётом эффективной ставки (APY с капитализацией),
выделяет скрытые условия и риск-флаги (лимит страхования АСВ 1 400 000 ₽, сгорание
процентов при досрочном закрытии, акционные и онлайн-условия) и **корректно отказывает**,
когда запрос невыполним или вне компетенции.

Образ результата — MVP полного жизненного цикла модели в продуктовой среде:
данные → ядро решения → объяснение LLM → валидация → мониторинг → противодействие
деградации → критерии вывода. Интерфейс — Streamlit ([`app/streamlit_app.py`](../app/streamlit_app.py),
вкладки Демо/Подбор/Качество/Мониторинг/О модели). Каталог — 29 учебных вкладов
([`data/deposits.csv`](../data/deposits.csv), штампуется датой сборки). Защита: презентация +
прототип + демо в контексте MLOps.

## 2. Почему гибрид «ядро + LLM» и управление сложностью ИИ-проекта

Решение разделено на **детерминированное ядро** ([`src/core`](../src/core)) — единственный
источник истины по всем цифрам — и **LLM-слой** ([`src/llm`](../src/llm), GigaChat-2-Max),
который только объясняет результат на естественном языке и **не вводит новых чисел**.
Ядро выполняет guardrails, фильтрацию, расчёт APY и ранжирование
([`src/core/recommender.py`](../src/core/recommender.py)); функция `recommend()`
детерминирована, не обращается к сети, задержка p95 ≈ 0,1 мс.

Управление сложностью означает, что недетерминированный компонент (LLM) выведен из
критического пути принятия решения. Контракт безопасности задан в
[`src/llm/explainer.py`](../src/llm/explainer.py): при `ACCEPT_THRESHOLD = 1.0` принимается
только текст, где **все** числа обоснованы числами ядра; иначе — откат к детерминированному
шаблону `render_template()` и фиксация инцидента. Без ключа GigaChat работает офлайн-режим
(детерминированные объяснения), поэтому всё демо воспроизводимо.

## 3. ML-solution-driven vs ML-model-driven

Проект — **solution-driven**: ценность создаёт целостное решение, а не одна предсказательная
модель. Критическое решение (что показать клиенту и когда отказать) принимает прозрачное
правило-ядро с управляемым модельным риском; LLM — генеративный слой UX, заменяемый
шаблоном без потери корректности. Это противоположность model-driven подходу, где качество
продукта целиком определяется метриками одной обученной модели. Архитектурное следствие:
галлюцинация LLM ведёт к действию `LLM_OFF` (отключить LLM, оставить ядро), а не к выводу
всего модуля — у системы есть безопасный детерминированный fallback.

## 4. ML-модель vs эконометрическая модель

ML-модель оптимизирует предсказание/ранжирование; эконометрическая — интерпретируемость и
идентификацию параметров. В «ВкладНавигаторе» ядро ближе к **прозрачной модели правил**:
эффективная ставка считается по формуле капитализации
([`src/core/effective_rate.py`](../src/core/effective_rate.py)), а ранжирование — по
фиксированным весам `GOAL_WEIGHTS` с версионированием `RANKING_VERSION = "ranking-1.0.0"`
([`src/core/ranking.py`](../src/core/ranking.py)). Объяснимость здесь — требование регулятора,
а не опция. LLM остаётся генеративным слоем поверх объяснимого ядра.

## 5. Где здесь causal inference и где его НЕТ

Подбор вклада — это **не причинный вывод**, а детерминированное ранжирование по формулам
(капитализация, веса цели). Пример: номинал 17,9% + ежемесячная капитализация → эффективная
(APY) 19,44% — ядро вычисляет это однозначно. Причинность важна **в оценке бизнес-эффекта**:
влияет ли ассистент на конверсию и нагрузку контакт-центра — вопрос A/B-эксперимента и
причинного эффекта, а не свойства самого ранжирования (см. Athey & Imbens, 2019 — ML-методы
для экономистов). Подробнее об эффекте — в [`docs/01_business_value.md`](01_business_value.md).
