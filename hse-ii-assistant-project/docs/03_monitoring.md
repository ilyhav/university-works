# 03. Мониторинг эксплуатации

Документ описывает критерий 3 проекта «ВкладНавигатор» (модуль подбора вклада):
**что и как мы наблюдаем** в работающем сервисе. Мониторинг — это контур обратной
связи MRM: он связывает реальный трафик с критериями вывода модели из эксплуатации
(`docs/06_decommission.md`) и опирается на детерминированное ядро как источник истины
(`docs/00_overview.md`, `docs/04_validation_model_risk.md`).

## Журнал ответов (JSONL, без ПДн)

Каждый ответ ассистента пишется одной плоской строкой в JSONL через
`src/monitoring/logging_store.py` (`log_response` → `response_to_record`).
**Персональные данные не сохраняются** — только параметры запроса и метрики ответа.
Поля записи:

- `ts` — метка времени UTC (ISO-8601);
- `ranking_version` (`ranking-1.0.0`), `status` (`ok`/`refused`), `refusal_code`
  (один из 8 кодов или `null`);
- параметры запроса: `goal`, `amount`, `term_months`, `currency`,
  `need_replenishment`, `need_withdrawal`;
- результат выдачи: `n_recommendations`, `top_id`, `top_effective_rate`,
  `top_income`;
- LLM-слой: `llm_used`, `groundedness_source`, `groundedness_score`,
  `ungrounded_count`;
- эксплуатация данных: `latency_ms`, `freshness_days`, `catalog_as_of`.

Журнал — единый источник для дашборда и расчёта дрейфа: его читают
`read_records` / `read_log_df`.

## Дашборд — вкладка «Мониторинг»

Вкладка `render_monitoring` в `app/streamlit_app.py` строит срез поверх журнала:

- **Объём запросов**: всего и разбивка «Запросы по дням» (`resample("D")`).
- **Доля отказов** и **причины отказов по кодам**: `value_counts` по
  `refusal_code` среди записей со `status == "refused"`.
- **Задержки**: средняя `latency_ms` (ядро ≈ 0,1 мс на p95 по валидации — порядок
  малых единиц).
- **Распределение сумм запросов**: гистограмма `amount` по 8 бинам.
- **Доля ответов с LLM**: среднее `llm_used`.
- **Дрейф входа (PSI: ранние vs недавние запросы)**: при ≥ 20 записях лог делится
  на трети, ранняя треть берётся за эталон, последняя — за текущее распределение;
  PSI считается по `amount` и `term_months` (`src/monitoring/drift.py`, `psi`,
  `psi_band`). Пороги классические: < 0,1 — нет дрейфа, 0,1–0,25 — умеренный,
  ≥ 0,25 — значимый.

Если журнал пуст, демо-трафик наполняется командой
`python -m scripts.demo_traffic`.

## Таблица контролируемых метрик

| Метрика | Источник | Порог / тревога | Действие |
|---|---|---|---|
| Доля отказов и коды | журнал, `refusal_code` | резкий рост одного кода | разбор причины, проверка guardrails |
| Точность ранжирования top-1 | `make eval` (golden) | `< 0.90` (critical `< 0.75`) | REFRESH каталога / разбор |
| Refusal precision | golden | `< 0.90` (critical `< 0.70`) | разбор анти-мисселинга |
| Refusal recall | golden | `< 0.85` | разбор пропусков отказа |
| Groundedness LLM | журнал, `groundedness_score`/`ungrounded_count` | `< 0.98` (critical `< 0.90`) | LLM_OFF (ядро остаётся) |
| Задержка p95 | журнал, `latency_ms` | `> 1500 мс` | расследование производительности |
| Дрейф входа PSI | дашборд, `psi` | `≥ 0.25` | пересмотр сценариев, перепроверка golden |
| Дрейф ставок PSI | `make drift`, `catalog_rate_psi` | `≥ 0.25` (рынок ушёл) | REFRESH каталога |
| Свежесть каталога | журнал, `freshness_days` | `> 14 дн.` (critical `> 28 дн.`) | REFRESH каталога |

## SLA свежести каталога (14 дней)

Свежесть = разница между сегодняшней датой и `as_of` каталога (каталог штампуется
датой сборки `make data`, поэтому свежесгенерированные данные свежи).
`freshness_status` (`src/monitoring/drift.py`)
сравнивает её с SLA = 14 дней: `≤ 14` — `ok`, `≤ 28` — `warning`, `> 28` —
`critical`. На вкладке «Подбор» при устаревании показывается предупреждение, при
критическом устаревании — ошибка о возможном несоответствии рынку. Сценарий дрейфа
(`make drift`, `scripts/simulate_drift.py`): при снижении ключевой ставки ЦБ с 16,0
до 11,0 (−5 п.п.) PSI ставок ≫ 0,25, MAE ставки ≈ 5,4 п.п., точность top-1
относительно актуального рынка падает 1.0 → 0.8, свежесть 107 дней (critical) →
вердикт **REFRESH**; после обновления каталога — top-1 = 1.0, вердикт **OK**.

## Связь с критериями вывода модели

Метрики дашборда — вход для `src/monitoring/decommission.py` (`THRESHOLDS`,
`evaluate`). Действия по нарастанию: **NONE → MONITOR → REFRESH → LLM_OFF →
DECOMMISSION**. Ключевая идея управления модельным риском: решение принимает
прозрачное ядро (объяснимость как требование регулятора — ЦБ РФ, «Обзор практик
применения ИИ в финансовом секторе», 2025), а LLM не находится в критическом пути.
Поэтому деградация groundedness ведёт к **LLM_OFF** (отключаем объяснения, оставляем
детерминированный fallback), а не к полному выводу модуля; **DECOMMISSION**
резервируется для отказа самого ядра. Подбор вклада — это ранжирование по формулам,
а не причинный вывод: причинность (A/B, эффект на конверсию; Athey & Imbens, 2019)
относится к оценке бизнес-эффекта, а не к контуру мониторинга качества выдачи.

См. также `docs/04_validation_model_risk.md` и `docs/06_decommission.md`.
