## Слайд 9. ML-специфичная хрупкость и как мы с ней боремся (критерии 4–5) — **[У2]**

> Это **главный слайд по требованию курса**: преподаватель ждёт не только прототип, но и
> анализ ML-специфичной «хрупкости» и способов её митигирования. Слайд отвечает на это прямо.

**На слайде:** таблица «ML-хрупкость → почему это не обычный баг → как ловим → как боремся» +
крупный тезис «ML-риск вынесен из критического пути решения».

**Текст (У2):**
«Это ключевой слайд. ML-специфичная хрупкость — это **не** падения и не исключения: их ловит
обычное тестирование. ML-система ломается **тихо** — модель уверенно выдаёт правдоподобно-
неверное, галлюцинирует, а качество дрейфует без единой ошибки в коде. Наша стратегия номер
один — **архитектурная**: мы осознанно вынесли принятие решения из ML в детерминированное
ядро, а LLM оставили вне критического пути. Это само по себе митигирование — большую часть
ML-риска мы просто убрали из контура решения. Остаточная ML-хрупкость живёт в двух местах: в
LLM-слое и в дрейфе данных. И под каждый её вид у нас — наблюдаемая метрика и конкретное
действие: галлюцинация → groundedness-проверка и откат к шаблону; prompt injection →
guardrail; дрейф рынка → PSI и SLA свежести с вердиктом REFRESH; сдвиг входа → честный отказ
вместо угадывания; тихая деградация ранжирования → golden-набор с независимым эталоном.»

| ML-специфичная хрупкость | Почему это не обычный баг | Как ловим (детект) | Как боремся (митигирование) |
|---|---|---|---|
| Галлюцинация LLM (выдуманная ставка) | модель **уверенно** выдаёт правдоподобно-неверное; юнит-тесты этого не видят | groundedness: каждое число из текста сверяется с ядром | `ACCEPT_THRESHOLD=1.0` → откат к шаблону; рост доли → `LLM_OFF` |
| Prompt injection | вход на естественном языке — вектор атаки, которого нет в обычном ПО | guardrail-классификатор по входному тексту | код `PROMPT_INJECTION`; текст юзера = данные; LLM вне критпути |
| Дрейф данных / концепт-дрейф | качество тихо падает; «правильный ответ» сам меняется во времени | PSI по ставкам + SLA свежести каталога | вердикт `REFRESH` → пересборка каталога (`make data`) |
| Сдвиг распределения входа | ML деградирует на «хвостах», которых не видел | PSI входа в мониторинге + проверка scope | отказы `OUT_OF_SCOPE` / `CONSTRAINTS_UNAVAILABLE` вместо догадки |
| Тихая деградация ранжирования | нет краша — просто хуже советы; не видно без эталона | golden-набор с **независимым** эталоном (перебор по APY) | регресс на golden + порог `ranking_top1 ≥ 0.90` |
| Завышенная уверенность / мисселинг | модель «продаёт» нереалистичное ожидание | сверка `expected_rate` с рынком | код `UNREALISTIC_EXPECTATION` |
| Невоспроизводимость / неаудируемость | ML-системы трудно повторить и проверить регулятору | версия `RANKING_VERSION`, фикс. сид, 42 теста | детерминированное ядро + версионируемый конфиг правил |

**Если спросят «где вообще ML-риск, если ядро детерминированное?»** — именно в этом смысл:
мы **перенесли** риск туда, где им можно управлять. ML-часть (LLM) не принимает решений, а
«обучаемая» часть, влияющая на выдачу (конфиг ранжирования + каталог), версионируется и
проверяется на golden-наборе. Это осознанный *ML-solution-driven* выбор, а не отсутствие ML.

**Демо-действие:** вкладка «О модели» — таблица «ML-хрупкость → митигирование».

**На слайде:** causal inference vs предсказание · ML vs эконометрика · ML-solution-driven ·

**Текст (У2):**

прозрачное ядро делает арифметику, а LLM — естественный язык. Это и есть управление модельным

## Частые вопросы комиссии (заготовки ответов)

- **«Почему не реальные данные банков?»** Проект — прототип (по ТЗ — учебный набор
  условий). Вымышленный каталог привязан к ключевой ставке ЦБ и воспроизводим одной
  командой; это важнее для MLOps, чем нестабильный парсинг живых сайтов. Структура данных
  при этом полностью реалистична.
- **«Где здесь машинное обучение, если ядро — формулы?»** Это осознанный
  ML-solution-driven выбор. "Обучаемая" часть — версионируемый конфиг скоринга `GOAL_WEIGHTS`,
  проверяемый на golden-наборе; LLM — генеративный слой. В финансах прозрачность ядра — не
  упрощение, а требование к управлению модельным риском.
- **«Что если GigaChat выдаст неверную ставку?»** Не дойдёт до клиента: groundedness-проверка
  сверяет каждое число с данными ядра, при несовпадении — откат к шаблону и запись инцидента.
- **«Как вы поняли, что top-1 действительно лучший?»** Эталон считается независимо от
  ранжирования — прямым перебором по эффективной доходности. Совпадение 1.0 значит, что
  скоринг согласован с экономически верным ответом, а не подогнан под себя.
- **«В чём ML-специфичная хрупкость вашего решения и как вы с ней боретесь?»** (ключевой
  вопрос — см. слайд 9). Хрупкость ML — это не краши, а **тихая** деградация: галлюцинации,
  дрейф данных, сдвиг распределения входа, переуверенность. Стратегия №1 — архитектурная:
  решение принимает детерминированное ядро, ML (LLM) вне критического пути. На остаточные
  риски — по детектору и мере: groundedness (галлюцинации), guardrails (injection/мисселинг),
  PSI + SLA свежести (дрейф), golden с независимым эталоном (тихая деградация ранжирования),
  пороги `decommission` (когда выводить/частично отключать).

## Глоссарий терминов (на случай вопросов)

- **ML-специфичная хрупкость** — режимы отказа, характерные для ML-систем, которых нет в
  обычном ПО: модель не «падает», а тихо ошибается. Главные: галлюцинация, дрейф,
  сдвиг распределения, переуверенность, невоспроизводимость.
- **Галлюцинация (hallucination)** — LLM уверенно выдаёт несуществующий факт/число. У нас
  ловится groundedness-проверкой.
- **Groundedness (обоснованность)** — доля чисел в тексте LLM, подтверждённых данными ядра;
  `1.0` = модель не придумала ни одной цифры. Порог приёмки `ACCEPT_THRESHOLD=1.0`.
- **Дрейф данных / концепт-дрейф (data/concept drift)** — распределение данных или сама
  зависимость «вход→правильный ответ» меняются со временем (рынок ушёл от каталога).
- **PSI (Population Stability Index)** — мера расхождения двух распределений; `< 0.1` — нет
  значимого дрейфа, `0.1–0.25` — умеренный, `≥ 0.25` — значимый.
- **Сдвиг распределения входа (distribution shift)** — запросы выходят за пределы того, на
  чём система «обучена»; ответ — честный отказ, а не догадка.
- **Prompt injection** — попытка через текст пользователя переопределить инструкции модели.
- **Guardrails** — детерминированные проверки на входе/выходе (scope, инъекции, мисселинг).
- **Golden-набор** — размеченные эталонные кейсы для регрессионной проверки качества.
- **MRR (Mean Reciprocal Rank)** — средний обратный ранг правильного ответа в выдаче.
- **APY / эффективная ставка** — годовая доходность с учётом капитализации (сложный процент).
- **ЖЦМ** — жизненный цикл модели: данные → обучение/конфиг → валидация → деплой →
  эксплуатация → мониторинг → вывод.
- **Decommission / лестница действий** — формальные пороги и эскалация реакций:
  `NONE → MONITOR → REFRESH → LLM_OFF → DECOMMISSION`.
