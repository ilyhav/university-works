## 1. Подготовка перед защитой (за 5 минут до начала)

Выполнить по порядку — это наполняет данные и журнал, чтобы дашборд был «живой»:

```bash
make data                        # каталог 29 вкладов + golden + сценарии (свежая дата → вердикт OK)
python -m scripts.demo_traffic   # 180 запросов в журнал мониторинга (~19% отказов)
make app                         # запустить веб-форму → http://localhost:8501  (ОСТАВИТЬ ОТКРЫТЫМ)
```

Во **втором окне терминала** держать наготове (не запускать заранее):

```bash
make eval     # валидация на golden + вердикт (для слайда 8 / 12)
make drift    # демонстрация деградации → REFRESH → восстановление (для слайда 11)
```

> Важно: `make data` непосредственно перед защитой штампует каталог сегодняшней датой,
> поэтому `make eval` гарантированно покажет вердикт **OK** (свежесть = 0).

## 2. Подключение GigaChat-2-Max (LLM-слой) — опционально

**LLM не обязателен.** Без ключа ассистент работает в офлайн-режиме: все цифры, отказы,
валидация и дрейф идентичны, меняется лишь стиль объяснения (детерминированный шаблон
вместо живого текста GigaChat). Демо полностью функционально без ключа.

### Откуда взять токен

1. Зайти на **developers.sber.ru** (Sber Studio) → раздел **GigaChat API**.
2. Создать проект (для учёбы достаточно тарифа физлица — `GIGACHAT_API_PERS`).
3. Скопировать **Authorization Key** — это строка Base64 от `client_id:client_secret`
   (в кабинете она так и называется «ключ авторизации»).

### Куда вставить токен

```bash
cp .env.example .env        # создать локальный .env (он в .gitignore — в репозиторий не попадёт)
```

Открыть `.env` и заполнить:

```ini
GIGACHAT_CREDENTIALS=<вставить Authorization Key (Base64)>
GIGACHAT_SCOPE=GIGACHAT_API_PERS      # физлицо; для бизнеса — GIGACHAT_API_B2B / _CORP
GIGACHAT_MODEL=GigaChat-2-Max         # самая сильная модель
GIGACHAT_VERIFY_SSL=false             # локально обычно false (сертификаты Минцифры РФ)
GIGACHAT_TEMPERATURE=0.2              # низкая → меньше галлюцинаций
```

Перезапустить `make app`. Проверка, что ключ подхватился: **в шапке приложения** и на
вкладке «О модели» статус сменится с «офлайн-режим» на «LLM включён (GigaChat-2-Max)».

### Как принудительно вернуть офлайн (если на защите упала сеть)

```ini
ASSISTANT_LLM_DISABLED=true     # в .env — жёстко отключает LLM, остаётся только ядро
```

> Что бы ни показал GigaChat — клиент **никогда не увидит выдуманную цифру**:
> groundedness-проверка сверяет каждое число с ядром, при несовпадении ответ
> заменяется детерминированным шаблоном, а инцидент пишется в журнал.

## 3. Возможности прототипа (5 вкладок Streamlit)

| Вкладка | Что показывает / умеет |
|---|---|
| **🎬 Демо** | Гид-режим: 6 кнопок-пресетов в один клик (скрытое условие 19,44 %, лимит АСВ, 3 отказа, смена цели). Под результатом — подсказка «что показать комиссии». |
| **🔎 Подбор** | Свободная форма: сумма, срок, валюта, цель, пополнение/снятие, гибкий горизонт, ожидаемая ставка, свободный вопрос. + готовые сценарии из выпадающего списка. Выдаёт сравнительную таблицу top-N с эффективной ставкой и риск-флагами — либо корректный отказ. |
| **📊 Качество** | Прогон golden-набора (17 кейсов) → метрики (17/17, top-1, MRR, precision/recall отказов) + вердикт эксплуатации по формальным порогам. |
| **📈 Мониторинг** | Журнал эксплуатации: объём запросов по дням, доля отказов, задержки, причины отказов по кодам, распределение сумм, дрейф входа (PSI). |
| **🛡 О модели** | Бизнес-ценность, архитектура, **таблица ML-хрупкости → митигирование**, пороги вывода модели. |

Что прототип умеет «под капотом»: расчёт APY с капитализацией, ранжирование под 4 цели,
страхование АСВ (только RUB), 8 кодов корректного отказа, groundedness-контроль LLM,
PSI-детектор дрейфа, формальные критерии вывода модели.

## 4. Сценарий демо: когда что включать (привязка к слайдам)

Демо разбито на **два живых блока** — по одному на докладчика. Между ними слайды идут как обычно.

### Блок 1 — Участник 1 (после слайда 7, тема «продукт»)

1. **Слайд 6 → вкладка «🎬 Демо»:** нажать **«Скрытое условие: 17,9 % → APY 19,44 %»**.
   Показать колонку эффективной ставки. Затем **«Сумма выше лимита АСВ»** — показать риск-флаг.
2. **Слайд 7 → та же вкладка «Демо»:** нажать подряд три отказа —
   **«завышенное ожидание»** (`UNREALISTIC_EXPECTATION`), **«вопрос про крипту»**
   (`OUT_OF_SCOPE`), **«prompt injection»** (`PROMPT_INJECTION`). Подчеркнуть: отказ конкретный.
3. (по желанию) **вкладка «Подбор»** — ввести свой запрос вручную, показать гибкость формы.

→ Передача слова Участнику 2.

### Блок 2 — Участник 2 (слайды 8, 10, 11)

4. **Слайд 8 → вкладка «📊 Качество»:** показать таблицу метрик и вердикт **OK**.
5. **Слайд 10 → вкладка «📈 Мониторинг»:** показать наполненный журнал (объём, доля
   отказов, дрейф входа). Упомянуть: клики из блока 1 тоже попали сюда — поток живой.
6. **Слайд 11 → второй терминал:** запустить **`make drift`**. Показать PSI ≫ 0,25,
   свежесть 107 дн., падение top-1 1.0 → 0.8, вердикт **REFRESH**, затем восстановление
   до top-1 = 1.0 и **OK** после обновления каталога.
7. (по желанию, слайд 12) **`make eval`** в терминале — показать текущий вердикт **OK**.

### Резервный план (если упала сеть / нет ключа GigaChat)

Всё демо работает в офлайн-режиме: ядро, отказы, валидация и дрейф **не зависят** от LLM.
`make eval` и `make drift` воспроизводятся из терминала. Вкладки «Демо»/«Подбор» отдают
детерминированные объяснения. Ничего отключать вручную не нужно — приложение само
определяет отсутствие ключа.
