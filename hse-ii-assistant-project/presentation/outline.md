## Слайд 1. Титул

- «ВкладНавигатор» — ИИ-ассистент потребителя финансовых услуг, модуль **подбор вклада**.
- Курс ВШЭ «Прикладное применение методов ИИ»; формат — презентация + прототип + демо в контексте MLOps.
- Архитектурный тезис одной строкой: детерминированное **ядро** (источник истины по цифрам) + **LLM-слой** (только объяснение).
- Автор, дата защиты; ссылка на репозиторий и [`../README.md`](../README.md).
- Демо: открытый прототип Streamlit на вкладке «Подбор» (заставка перед стартом).

## Слайд 2. Задача и образ результата

- Проблема клиента: ставки выглядят похоже, но реальный доход искажают капитализация, акции, лимит АСВ, условия досрочного закрытия.
- Задача модуля: по параметрам (сумма, срок, валюта, цель, пополнение/снятие, ожидаемая ставка, свободный вопрос) выдать **сравнительную таблицу** с эффективной ставкой и риск-флагами — либо **корректно отказать**.
- Образ результата: top-N вкладов под цель + честные предупреждения + объяснение на естественном языке, обоснованное цифрами ядра.
- Границы: это не инвестсоветник; кредиты/акции/крипта/страховки — вне компетенции (отказ `OUT_OF_SCOPE`).
- Демо: показать форму ввода с предзаполненным типовым сценарием из `data/client_scenarios.json`.

## Слайд 3. Бизнес-ценность и эффект на процесс (критерий 1)

- Куда встраивается: точка выбора вклада (онбординг/контакт-центр/ДБО) — сейчас это ручное сравнение и риск мисселинга.
- Три направления эффекта: рост конверсии в открытие вклада, снижение нагрузки на контакт-центр, снижение регуляторного риска мисселинга (анти-мисселинг через отказ `UNREALISTIC_EXPECTATION`).
- Прозрачность как требование регулятора: все цифры детерминированы и воспроизводимы, объяснение нельзя «придумать» (groundedness).
- Как корректно мерить эффект: это **причинный** вопрос (A/B-тест на конверсию), а не просто предсказание — см. слайд 13 и [`../docs/01_business_value.md`](../docs/01_business_value.md).
- Демо: вкладка «О модели» — раздел бизнес-ценности и образа процесса.

## Слайд 4. Архитектура гибрида «ядро + LLM» (критерий целостности)

- Поток: Streamlit → **ядро** (`src/core`) → **LLM-слой** (`src/llm`) → **журнал** (`src/monitoring/logging_store.py`).
- Ядро — источник истины: guardrails (`guards.py`) → фильтрация (`filtering.py`) → эффективная ставка (`effective_rate.py`) → ранжирование (`ranking.py`, `RANKING_VERSION="ranking-1.0.0"`).
- LLM (GigaChat-2-Max, `gigachat_client.py`) **пересказывает** результат ядра и **не вводит новых чисел**; офлайн-режим без ключа даёт детерминированные объяснения.
- Anti-hallucination: `groundedness.py` + `explainer.py` (`ACCEPT_THRESHOLD=1.0`) — необоснованное число → откат к шаблону, инцидент в журнал.
- Демо: показать диаграмму архитектуры из [`../README.md`](../README.md); кратко — LLM не в критическом пути принятия решения.

## Слайд 5. ЖЦМ как MVP (критерий 2)

- Полный цикл одним конвейером: данные → ядро решения → объяснение LLM → валидация → развёртывание → мониторинг → противодействие деградации → критерии вывода.
- Воспроизводимость через Makefile: `make data` (каталог + сценарии + golden), `make app`, `make eval`, `make drift`, `make test` (42 теста).
- Слои репозитория = стадии ЖЦМ: `src/core → src/llm → src/validation → src/monitoring`; контейнеризация (`Dockerfile`, `docker-compose.yml`).
- Данные детерминированы: ставка = base(ключевая ЦБ, базовая 16.0) + надбавка за срок + надбавка архетипа; 29 вкладов, банки вымышленные, каталог штампуется датой сборки (`make data`).
- Демо: терминал — `make eval` запускается одной командой (разбор результата на слайде 9).
- Подробно: [`../docs/02_ml_lifecycle.md`](../docs/02_ml_lifecycle.md).

## Слайд 6. Демо подбора вклада

- Ввод типового сценария → ядро возвращает сравнительную таблицу top-N под цель клиента.
- Ключевой кейс «скрытое условие»: номинал **17,9 %** + ежемесячная капитализация → эффективная (APY) **19,44 %** — считает ядро, не LLM.
- Риск-флаги в таблице: превышение лимита страхования **АСВ 1 400 000 ₽**, сгорание процентов при досрочном закрытии, отсутствие капитализации, акционные/только-онлайн условия.
- Объяснение под таблицей обосновано числами ядра; при выключенном ключе — тот же текст в офлайн-режиме.
- Демо: задать сумму выше лимита АСВ, чтобы наглядно сработал флаг страхования; показать колонку «эффективная ставка».

## Слайд 7. Корректные отказы — 8 кодов (демо)

- Отказ — отдельный измеримый класс ответа, все проверки детерминированы и покрыты тестами (`src/core/guards.py`).
- Восемь кодов:

  | Код | Когда срабатывает |
  |---|---|
  | `INVALID_INPUT` | некорректный ввод (сумма ≤ 0, срок вне 1–120 мес. и т.п.) |
  | `OUT_OF_SCOPE` | вопрос вне подбора вклада (кредиты/акции/крипта/страховки) |
  | `UNREALISTIC_EXPECTATION` | ожидаемая ставка > рынок + 3 п.п. (анти-мисселинг) |
  | `AMOUNT_OUT_OF_RANGE` | сумма не подходит под доступные вклады |
  | `TERM_UNAVAILABLE` | в каталоге нет нужного срока |
  | `CURRENCY_UNAVAILABLE` | в каталоге нет вкладов в выбранной валюте |
  | `CONSTRAINTS_UNAVAILABLE` | нет вкладов под дополнительные условия |
  | `PROMPT_INJECTION` | попытка переопределить инструкции ассистента |

- Отказ всегда конкретный и полезный (подсказывает доступные сроки/минимальный взнос), а не «ничего не найдено».
- Демо: показать 2–3 отказа подряд — завышенное ожидание (анти-мисселинг), вопрос про крипту (вне компетенции), prompt injection.

## Слайд 9. ML-специфичная хрупкость и митигирование (ключевой слайд, критерии 4–5)

- Тезис крупно: **ML ломается тихо** (галлюцинации, дрейф, сдвиг входа, переуверенность) — это не краши, обычное тестирование их не ловит. **Стратегия №1 — вынести решение из ML в детерминированное ядро**, LLM вне критического пути.
- Таблица «ML-хрупкость → почему не обычный баг → детект → митигирование»:

  | ML-хрупкость | Детект | Митигирование |
  |---|---|---|
  | Галлюцинация LLM | groundedness (число vs ядро) | откат к шаблону; рост доли → `LLM_OFF` |
  | Prompt injection | guardrail по входу | код `PROMPT_INJECTION`; LLM вне критпути |
  | Дрейф / концепт-дрейф | PSI ставок + SLA свежести | вердикт `REFRESH` → `make data` |
  | Сдвиг входа | PSI входа + scope | отказы `OUT_OF_SCOPE` / `CONSTRAINTS_UNAVAILABLE` |
  | Тихая деградация ранжирования | golden с независимым эталоном | порог `ranking_top1 ≥ 0.90` |
  | Мисселинг (переуверенность) | `expected_rate` vs рынок | код `UNREALISTIC_EXPECTATION` |
  | Невоспроизводимость | `RANKING_VERSION`, фикс. сид | детерминированное ядро + 42 теста |

- Методология MRM: ЦБ РФ «Обзор практик применения ИИ в фин. секторе» (2025), SR 11-7, KPMG/Deloitte (источники риска: данные / модель / использование).
- Готовый ответ на каверзный вопрос «где ML-риск, если ядро детерминированное?» — риск перенесён туда, где им можно управлять (см. текст доклада).
- Демо: вкладка «О модели» — таблица «ML-хрупкость → митигирование»; подробно [`../docs/04_validation_model_risk.md`](../docs/04_validation_model_risk.md).

## Слайд 10. Мониторинг и дашборд (критерий 3)

- Каждый ответ пишется в журнал JSONL (`src/monitoring/logging_store.py`) → агрегация в DataFrame.
- Что видно на дашборде (вкладка «Мониторинг»): объём запросов, доля отказов, задержки, дрейф входа (PSI), свежесть каталога.
- Наполнение демо-трафиком: `python -m scripts.demo_traffic` — журнал готов к показу на защите.
- Метрики мониторинга напрямую питают критерии вывода модели (слайд 12): один снапшот → один вердикт.
- Демо: открыть вкладку «Мониторинг» с уже наполненным журналом; подробно [`../docs/03_monitoring.md`](../docs/03_monitoring.md).

## Слайд 11. Противодействие деградации — `make drift` (критерий 5)

- Сценарий рынка: ключевая ставка **11.0** (−5 п.п.), промо-надбавка −3, max +0.5 — каталог рассинхронизировался с рынком.
- Что показывает детектор: дрейф ставок **PSI ≫ 0,25** (значимый), MAE по ставке ≈ **5,4 п.п.**, свежесть **107 дней** (critical).
- Эффект на качество: точность top-1 относительно актуального рынка **1.0 → 0.8** → вердикт **REFRESH**.
- Восстановление: после обновления каталога — **top-1 = 1.0**, вердикт **OK** (полный цикл деградация → реакция → восстановление).
- Демо: терминал — `make drift`, показать переход REFRESH → OK; подробно [`../docs/05_degradation.md`](../docs/05_degradation.md).

## Слайд 12. Критерии вывода модели из эксплуатации (критерий 6)

- Решение по формальным порогам (`src/monitoring/decommission.py`, `THRESHOLDS`), а не «на глаз».
- Пороги: groundedness ≥ 0.98 (crit. 0.90); refusal precision ≥ 0.90 (crit. 0.70); refusal recall ≥ 0.85; ranking top-1 ≥ 0.90 (crit. 0.75); latency p95 ≤ 1500 мс; свежесть warning 14 / critical 28 дней.
- Лестница действий по нарастанию: **NONE → MONITOR → REFRESH → LLM_OFF → DECOMMISSION**.
- Ключевая развилка: есть безопасный детерминированный fallback, поэтому **галлюцинации LLM → `LLM_OFF`** (оставляем ядро), а не полный вывод модуля.
- Демо: вкладка «О модели» / результат `make eval` — текущий вердикт **OK**; подробно [`../docs/06_decommission.md`](../docs/06_decommission.md).

## Слайд 13. Теоретический контекст курса

- **Causal inference vs предсказание:** подбор вклада — это ранжирование по детерминированным формулам, а **не** причинный вывод; причинность нужна для оценки бизнес-эффекта (A/B, эффект на конверсию). Athey & Imbens (2019) — ML-методы для экономистов.
- **ML-модель vs эконометрика:** ML оптимизирует предсказание/ранжирование, эконометрика — интерпретируемость и идентификацию параметров; здесь ядро ближе к прозрачной модели правил (объяснимость как требование регулятора), LLM — генеративный слой UX.
- **ML-solution-driven vs ML-model-driven:** решение принимает детерминированное ядро (управляемый модельный риск), LLM не в критическом пути → управление сложностью.
- **Модельный риск:** MRM-подходы (ЦБ РФ 2025, SR 11-7, KPMG, Deloitte) — источники риска данных/модели/использования, валидация, мониторинг.
- Демо: слайд без демо; ссылка на [`../docs/00_overview.md`](../docs/00_overview.md).

## Сценарий демонстрации (последовательность действий на защите)

Подготовка до начала (не на экране): `make data`, затем `python -m scripts.demo_traffic`
для наполнения журнала мониторинга; запустить `make app` (http://localhost:8501).

1. **Подбор (слайд 6).** Открыть вкладку «Подбор», загрузить типовой сценарий. Показать сравнительную таблицу top-N, колонку эффективной ставки и кейс «номинал 17,9 % → APY 19,44 %». Задать сумму выше **1 400 000 ₽** — сработает риск-флаг АСВ.
2. **Отказы (слайд 7).** Прогнать три отказа подряд: завышенное ожидание ставки (`UNREALISTIC_EXPECTATION`, анти-мисселинг), вопрос про крипту/акции (`OUT_OF_SCOPE`), фразу с «ignore previous instructions» (`PROMPT_INJECTION`). Подчеркнуть, что отказ конкретный.
3. **Качество (слайды 8–9).** Перейти на вкладку «Качество»: метрики golden-набора (17/17, top-1 = 1.0, отказы F1 = 1.0, confusion tp8/fp0/fn0/tn9). Связать с таблицей мер модельного риска.
4. **Мониторинг (слайд 10).** Открыть вкладку «Мониторинг» с наполненным журналом: объём, доля отказов, задержки, дрейф входа, свежесть.
5. **Деградация (слайд 11).** В терминале запустить `make drift`: показать PSI ≫ 0,25, свежесть 107 дней, падение top-1 1.0 → 0.8, вердикт **REFRESH**, затем восстановление до top-1 = 1.0 и **OK** после обновления каталога.
6. **Вывод модели (слайд 12).** Показать результат `make eval` / вкладку «О модели» с текущим вердиктом **OK** и пояснить лестницу NONE → MONITOR → REFRESH → LLM_OFF → DECOMMISSION (галлюцинации → `LLM_OFF`, не полный вывод).
7. **Резерв (офлайн-режим).** Если нет ключа GigaChat — подчеркнуть, что всё демо работает в офлайн-режиме на детерминированных объяснениях; числа и отказы не меняются.

Резервный план при отказе сети/ключа: вести демо полностью в офлайн-режиме (ядро не зависит от LLM), результаты `make eval` и `make drift` воспроизводятся из терминала.
